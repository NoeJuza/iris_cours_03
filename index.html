Préambule: Terminal et lignes de comandes

Pour executer des lignes de commandes dans windows, on va utiliser l'invite de commande qui se lance en appuyant sur la touche windows pour lancer le menu démarrer puis il suffit de taper "cmd" afin d'avoir comme proposition un programme appelé "invite de commande"

Alternativement, on peut lancer un terminal (invite de commande) directement dans VSCode et appuyant sur le menu "Terminal" dans la barre en haut, puis en cliquant sur "new terminal"

Quelques commandes de base windows:

  • echo qqch permets d'imprimer "qqch" dans le terminal
  • echo %cd% permets d'imprimer le répertoire courrant dans le terminal
  • dir permets d'afficher les éléments présents dans le répertoire courrant
  • cd chemin permets de changer de répertoire vers le sous-répertoire "chemin", alternativement on peut préciser un chemin absolu pour s'y rendre comme par exemple cd C:\Users
  • mkdir "nom de dossier" permets de créer un nouveau dossier nomé nom de dossier dans le dossier courrant

VCS? Git?

Un VSC (Version Control System) sers à suivre les évolutions d'un ou plusieurs fichiers dans le cours du temps. La problématique à laquelle réponds les solutions de VSC est le problème des versions de fichier, le fait de devoir recréer un fichier _v1 _v2 _v3 ... _vfinale. Avec un VCS on n'a plus besoin de créer plein de versions de fichiers différents, on peut garder un seul projet qu'on modifie au cours du temps.

Git c'est un VSC, probablement LE VSC le plus populaire, développé initiallement par le même développeur que le noyau linux, aujourd'hui un projet open-source sur lequel travaille énormément de gens

Comment ça marche?

Dans git on apellera l'ensemble des fichiers traqués par celui-ci un "dépot" (repository), un dépot se constitue de plusieurs fichiers et les informations relatives au dépot sont stoqués dans un fichier ".git" à la racine de l'arborescence de fichiers du projet

Git fonctionne avec un concept "d'arbre" pour traquer les changements apportés à un projet, chaque "noeud" de l'arbre représente un "commit" qui est en gros un "paquet de modifications" aportées par le développeur au projet. Dans git on a quelque chose de très intéressant étant donné que les modifications sont stockées sous forme d'arbre c'est qu'on peut "forker" un dépot (repository) git c'est à dire prendre un dépot existant et créer une "copie" à nous la fusionner plus tard avec le dépot initial afin d'ajouter nos changements au projet.

Git est un outil qui s'utilise principalement en "ligne de commande" (dans le terminal) les commandes les plus importantes sont:

  • git init . permets d'initialiser un dépot (repository) dans le dossier courrant (il s'occupe de créer un ".git" avec des infos de base
  • git checkout -B nomDeLaBranche permets de changer la branche courrante (et de la créer si besoin) vers "nomDeLaBranche (la branche par défaut se nomme "main"), en général on va créer une branche par nouvelle fonctionalité dans le projet
  • git branch -D nomDeLaBranche permets de supprimer la branche avec le nom "nomDeLaBranche"
  • git add . ajoute tous les changements du répertoire courrant a git et les mets dans la partie "staging"
  • git commit -am "description du travail effectué" permets de créer un "commit" appelé "description du travail effectué" et ajoute donc un nouveau noeud à la branche courrante

Plein d'autres commandes sont disponibles sur la documentation oficielle

GitHub alors?

Si un dépot git représente une échope, GitHub c'est un grand marché, c'est un endroit centralisé ou chaqun peut stocker ses et partager ses repos et ses projets avec git. Accessible n'importe ou, GitHub est utile en tant que solution pour les projets ou on travaille à plusieurs ou même juste pour permettre à n'importe qui d'accéder à nos codes depuis n'importe ou.

De nos jours, le compte GitHub fait partie intégrante du CV d'un développeur au même titre qu'un compte linkedin ou autre, le compte github permets de voir comment un développeur écrit du code, si il participe à beaucoup de projet, travaille sur des projets personnels en dehors du travail ou des cours et ainsi de suite.

Repo local VS distant

Pour le moment nous avons uniquement de la partie "locale" de git, afin d'avoir un code accessible par n'importe qui sur github il faut configurer notre dépot local pour qu'il puisse intéragir avec un dépot "distant", pour cela on a quelques commandes pertinantes:

  • git remote add nomDeLaSourceDistante lienversLeRepoDistant permets d'ajouter une source distante nommée "nomDeLaSourceDistante" qui pointe vers le lien passé en paramètre, en général le nom de la source distante sera "origin" et le lien sera un lien github du type: "https://github.com/user/project.git"
  • git push nomDeLaSourceDistante nomDeLaBrancheDistante permets de "pousser" votre dernier commit local de la branche courrante vers le repo distant et la branche selectionnée (pour des projets relativement simples ça sera "main" au début si on est seul sur le projet)
  • git pull nomDeLaSourceDistante permets de récupérer la dernière version du code disponible sur le dépot distant
